## 1. 產品一句話

在 Chrome 瀏覽器中，對**正在播放的影片／音訊**提供**即時或可近即時**的**翻譯字幕（必要時含配音）**，尤其補足**無內建字幕**的內容。

---

## 2. 要解決的問題

| 痛點 | 說明 |
|------|------|
| 語言隔閡 | 使用者想理解外語影片、直播、課程，但沒有字幕或字幕語言不對 |
| 工具碎片化 | 需切換 App、上傳檔案、或僅支援少數網站 |
| 信任與成本 | 音訊是否上雲、費用模式、延遲與準確度不透明 |

---

## 3. 目標使用者（可複選，待確認）

- **A. 一般閱聽者**：YouTube／社群／新聞影片，偶發需求，在意簡單與低費用  
- **B. 學習者**：長時間觀看，在意可讀性、術語、是否可對照原文  
- **C. 工作者**：會議錄影、線上課程、內部站台，在意隱私與合規（常需**企業／自架**路線）  

**請確認**：首發以哪一類為**Primary**？（建議先鎖 1 類，避免 MVP 發散。）

---

## 4. 價值主張（草案）

1. **站內完成**：不離開瀏覽器分頁即可看見翻譯結果（字幕浮層或配音）。  
2. **無字幕也可啟用**：技術路線以「分頁音訊 → 辨識 → 翻譯 → 呈現」為核心（實際延遲與品質依引擎而定）。  
3. **權限與隱私可溝通**：清楚說明音訊處理在**本機／自架／雲端**何處發生（上架與信任門檻）。

---

## 5. 產品範圍分層

### 5.1 MVP（建議定義，待你勾選／修改）

| ID | 能力 | 說明 |
|----|------|------|
| M1 | 手動啟停 | 使用者於 popup 主動「開始／停止」擷取（符合 Chrome 使用者手勢與 `tabCapture` 慣例） |
| M2 | 單一分頁對應 | 僅對**當前啟用擷取的分頁**顯示狀態與字幕浮層（避免全瀏覽器干擾） |
| M3 | 字幕浮層 | 可調位置／字級／對比（**至少**固定底部浮層 + 基本可讀性；進階樣式可 Phase 2） |
| M4 | 一條 STT 管線 | **本版實作（v0.7.x+）：本機 whisper-server（Whisper 類）**；長期草案仍允許未來另支援雲端串流 ASR |
| M5 | 一條 MT 管線 | 來源語言可「自動偵測」或手選；目標語言於設定中選擇 |
| M6 | 隱私與說明 | 專用「隱私與資料處理」說明頁 + 權限理由與 Chrome Web Store 文案 |

### 5.2 Phase 2（暫列，不綁 MVP）

- 多網站捷徑（例如偵測 `video` 元素一鍵啟用）  
- 原文＋譯文對照、暫存本機歷史（可關閉）  
- 配音（TTS）與音量混音策略  
- 付費方案／用量顯示／API Key 由使用者自行填入  

### 5.3 明確「不做」或「暫不做」（降低誤期）

- 不宣稱「零延遲」；以**可量測的端到端延遲區間**溝通  
- 不預設支援 DRM 受保護內容（多數情境**無法**擷取音訊，需 UX 說明）  
- 不將「全網站 100% 相容」列為 MVP 承諾

## 6. 核心使用者流程（文字版線框）

```
開啟有影片的一般網頁
    → 點擊擴充圖示（popup）
    → [開始擷取音訊]（必要時提示：需有聲音播放、部分網站不支援）
    → 頁面出現「擷取中／連線中／辨識中」狀態
    → 浮層顯示譯文（可選：片段原文）
    → [停止] 釋放擷取與連線
```

**例外流程（需在 UI 說明）**

- `chrome://`、`edge://`、部分內建頁：無法擷取  
- 無聲音／靜音分頁：提示檢查分頁音量  
- DRM／網站政策：提示可能不支援

## 7. 體驗原則（設計約束）

| 原則 | 具體含義 |
|------|----------|
| 可預期 | 何時開始上傳／處理音訊、費用與連線狀態一眼可見 |
| 不打斷播放 | 浮層不攔截影片點擊（`pointer-events` 策略與可拖移區域分離） |
| 可關閉 | 一鍵停止並清除畫面上狀態 |
| 最小權限 | `host_permissions` 與 `matches` 隨產品策略收斂（上架加分） |

---

## 8. 技術與合規邊界（產品需知情）

| 項目 | 與產品設計的關聯 |
|------|------------------|
| `tabCapture` | 需使用者手勢啟動；與 popup 流程綁定 |
| Offscreen / AudioContext | 音訊處理適合放 offscreen；延遲與 CPU／GPU 影響體感 |
| 雲端 STT／MT | 涉及傳輸與帳務；需隱私政策與選用區域／供應商說明 |
| 本機／自架 | 安裝門檻高，但利於 B2B／隱私敏感族群 |
| Web Store 審查 | 權限理由、`narrow host permissions`、資料使用揭露需與實作一致 |

## 9. 成功指標（建議，待你定 KPI）

| 類型 | 範例指標 |
|------|----------|
| 可用性 | MVP 在指定測試網站（例：公開 HTML5 影片頁）**連續 10 分鐘**不中斷 |
| 體感 | 端到端字幕延遲中位數 & P95（依引擎訂目標，例如先訂「可讀」再優化） |
| 品質 | 指定片段的**字級可讀率**主觀評分或簡單 BLEU／人工抽測（若走 MT） |
| 信任 | 商店差評中「隱私／權限」占比維持低於某閾值 |

## 10. 風險與假設

| 風險 | 緩解方向 |
|------|----------|
| 延遲高於使用者預期 | MVP 先標註「Beta／實驗」、顯示連線與緩衝狀態 |
| 網站不相容 | 維護「已知限制」清單與錯誤訊息 |
| 雲端成本 | 用量上限、BYOK、或僅限付費 |
| 法遵／著作與服務條款 | 產品聲明「不規避 DRM」、提醒使用者遵守內容平台規範 |

**假設（若錯誤需回頭改範圍）**

- 首發以 **Chrome（Chromium）** 為主；Edge 可列相容待測  
- 使用者接受「手動啟停」作為 v0 互動模型

## 11. 待你確認清單（請逐項回覆或勾選）

1. **Primary 使用者**：A／B／C 何者為主？是否允許「僅英文介面」首發？  
2. **MVP 語言對**：首發必須支援的「來源 → 目標」組合？（例：`en → zh-TW`）  
3. **STT 偏好**：雲端 API（易上線）vs 本機／自架（隱私）— 是否兩條路都要在路線圖寫死？  
4. **呈現形式**：MVP 僅字幕，還是**必須**含 TTS 配音？  
5. **商業模式**：免費試用／訂閱／BYOK／完全開源— 是否已有方向？  
6. **相容範圍**：MVP 是否只保證「一般 HTML5 `video` ＋ 非 DRM」？是否要把 YouTube／Netflix 列為明示目標（將影響測試與溝通）？  
7. **資料留存**：是否**禁止**儲存音訊；逐字稿是否允許僅存本機、可一鍵清除？
